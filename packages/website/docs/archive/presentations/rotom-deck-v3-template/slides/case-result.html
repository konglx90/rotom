设计原则 1 · rotom 不另搞一套 · 全部复用已有能力
  • 群 = 普通群 + 一个标记 · 巡检 = groups.type=patrol(src/master/api/groups.ts)· 跨机器部署 = group_member_settings 覆盖(src/master/group-paths.ts)· 没有专用 UI,没有专用路由,没有专用协议
  • 主动出击 = 定时任务派 issue · scheduled_tasks.handler_key=issue-patrol(src/master/scheduler-handlers.ts)· 共用 30s tick + at-most-once + grace window + dashboard 可见
  • 沉淀资产 = 已有 memory / skill / issue · 巡检规则走 issue-patrol-rules skill(rotom 内已建 · /api/skills);memory 走 memory add;产物走 /api/artifacts · 全部 rotom 已有
设计原则 2 · 写盘必挂 issue · 让 audit trail 自然形成
  • 硬规则写在 skill 里 · skill/rotom-a2a-communicate/SKILL.md 写死"写盘必挂 issue",启动时写到 ~/.rotom/SKILL.md(src/shared/skill-md.ts)· Agent 拿不到绕过
  • 所有写盘操作都进 issue_events 表 · issue 创建 / 派发 / 审批 / 中断 / 完成 / 失败全留痕(src/master/db/issues.ts + issue_events)· dashboard 可回放
  • 审批粒度可选 · approval_policy ∈ {r_allow, rw_allow}(migration 029)· r_allow 下写盘走 PreToolUse hook 挂 dashboard 审批 · 只读 Bash 走 src/shared/readonly-allowlist.ts 内置白名单(fail-closed)
设计原则 3 · 群聊 = 工作台 · @ 路由 + 并发 + 自我纠错
  • @ 谁谁接活 · src/master/router.ts + ws-hub/routing.ts · Agent 之间 @ 也走同一路径 · 2fec433e 群江德福@瓦力接力 review = 群消息驱动 agent 切换
  • #reply 触发桥接 + 5min 升级 · ask_bridges 表(src/master/db/ask-bridges.ts)+ ask-bridge-check handler(20s tick)· A @B + #reply → B 不回 5min 自动建 issue 升级
  • 多群并发 = 单 Agent maxConcurrent(默认 2),不是全局 · src/executor/worker.ts · 西花同时在 c211307e 放任务 + 2fec433e 排查,江德福两边接活互不堵
  • Agent 也会被实测打脸 + 用硬数据重做 · 2fec433e 群:瓦力 07:26 判"非对象 TypeError",西花 07:32/07:33 两次实测反驳,瓦力用 V8 实测纠错 + 三方证据关掉 R2 风险 · 单 Agent 给不出这种"被打脸后用硬数据重做"的回路
设计原则 4 · 持续学习 = 沉淀的资产在 rotom 里
