背景 · 基础交互05 / 18

群就是工作台 · @ 交互 + 可以并发

Dashboard 6 块都挂在群下面,那「群本身」就是工作台 —— 人和 Agent 怎么用它?三件事:@ 谁谁接活、#reply 触发桥接、多群并发。

@ 谁谁接活 · 路由到 worker

群消息里 @西花 / @江德福 / @瓦力 → Master ws-hub/routing 按 @ 解析目标 → 走 router.ts 派给对应 Worker(不是 broker,不直接 client→client)。Agent 之间互相 @ 也走同一路径(江德福 @ 瓦力接力 review 就是这条)。

代码:src/master/ws-hub/routing.ts + src/master/router.ts

#reply 触发桥接 · 5min 不回自动升级

A 在群里 @B + 内容里带 #reply 标记 → Master 自动建 ask-bridge(ask_bridges 表)。B @ 回复 A → mark answered;B 离线/不 @ 回复 → scheduler handler ask-bridge-check(20s tick)兜底 → 5min 仍无解 → 建 Issue 升级。

代码:src/master/db/ask-bridges.ts + scheduler-handlers.ts

多群并发 · 人同时管多件事的体感

Master 一个进程,接所有群的 WS 流量,谁在哪个群都无感。Issue 抢单 maxConcurrent(默认 2)限的是单 Agent 并发,不是全局 —— 西花在群 A 让江德福跑 issue,群 B 同时让瓦力跑 issue,两个 Agent 同时推进。同一群内多消息按群 worker-chat 串行队列。

代码:src/executor/worker.ts + worker-chat.ts

Master 自己处理路由 / 排队 / 限流

人不用关心并发:Master 做消息去重(src/shared/dedup.ts·5min TTL)、限流(默认 60 msg/min/agent · 豁免 chunk/heartbeat)、离线队列(100 条/24h TTL)。所有数字员工 不直连,全部经 Master 中转 —— 没有点对点连接。

代码:src/master/offline-queue.ts + src/shared/dedup.ts

外部集成 · 嵌入其他工作流 · 与 mesh Agent 沟通

你的 codex / claude 在外面干别的活,中间卡了 —— 把 mesh 里的 agent 拉进来问一句就完事。前提:Master :28800 已跑 + token 在 ~/.rotom/executor.config.json。

Client 端装一次(每台接入的机器)
① tnpm install -g @alipay/rotom
② rotom init --master 10.x.x.x:28800 --tools claude,codex,hermes,openclaw
③ 写 ~/.rotom/executor.config.json(worker 启动参数 + mesh token + Master 地址)

两种群模式(按场景二选一)

普通群 · 多 Agent 协作 · @ 触发 worker 自动接话

→ rotom group create "<需求>" --agents a,b,c --message "@全体 ..."

单播群 (--a2a-direct) · 点对点 · 消息不广播 · worker 不自动醒 · 只在 --need-reply 显式点名时起一手

→ rotom group create "<你和 B 的小窗口>" --agents 你,B --a2a-direct

沟通循环 · 一问一答

提问:rotom group send <gid> <target> "<问题>" --need-reply

master 自动补 @target + 登记 requestId → asker;对方回复时 master 硬剥 @<asker> 再入库,你的 worker 不被触发,不会来回聊

等回复:poll-replies.sh <gid> --as <你> · 默认 10 轮 × 30s = 5min · 找到退出 0 · 超时退出 2

能力 2 · 主动出击 / 复用 群 + 定时任务 + Issue

主动出击不是另搞一套 —— 复用 rotom 三个已有的能力(群 / 定时任务 / Issue)拼出来。建 1 个 type=patrol 群,挂 1 个每小时跑的定时任务,任务就是建 1 个 [巡检] issue 让巡检员自己判。

三个已有能力怎么拼(没有新概念)

1
复用「群」 · 巡检群 = 普通群 + type=patrol 标记(groups.type 字段)· 限 1 个未归档 · 限 1 个巡检员 · 走群消息 + issue 完整流程,没有任何专用 UI
2

设计原则 1 · rotom 不另搞一套 · 全部复用已有能力
  • 群 = 普通群 + 一个标记 · 巡检 = groups.type=patrol(src/master/api/groups.ts)· 跨机器部署 = group_member_settings 覆盖(src/master/group-paths.ts)· 没有专用 UI,没有专用路由,没有专用协议
  • 主动出击 = 定时任务派 issue · scheduled_tasks.handler_key=issue-patrol(src/master/scheduler-handlers.ts)· 共用 30s tick + at-most-once + grace window + dashboard 可见
  • 沉淀资产 = 已有 memory / skill / issue · 巡检规则走 issue-patrol-rules skill(rotom 内已建 · /api/skills);memory 走 memory add;产物走 /api/artifacts · 全部 rotom 已有
设计原则 2 · 写盘必挂 issue · 让 audit trail 自然形成
  • 硬规则写在 skill 里 · skill/rotom-a2a-communicate/SKILL.md 写死"写盘必挂 issue",启动时写到 ~/.rotom/SKILL.md(src/shared/skill-md.ts)· Agent 拿不到绕过
  • 所有写盘操作都进 issue_events 表 · issue 创建 / 派发 / 审批 / 中断 / 完成 / 失败全留痕(src/master/db/issues.ts + issue_events)· dashboard 可回放
  • 审批粒度可选 · approval_policy ∈ {r_allow, rw_allow}(migration 029)· r_allow 下写盘走 PreToolUse hook 挂 dashboard 审批 · 只读 Bash 走 src/shared/readonly-allowlist.ts 内置白名单(fail-closed)
设计原则 3 · 群聊 = 工作台 · @ 路由 + 并发 + 自我纠错
  • @ 谁谁接活 · src/master/router.ts + ws-hub/routing.ts · Agent 之间 @ 也走同一路径 · 2fec433e 群江德福@瓦力接力 review = 群消息驱动 agent 切换
  • #reply 触发桥接 + 5min 升级 · ask_bridges 表(src/master/db/ask-bridges.ts)+ ask-bridge-check handler(20s tick)· A @B + #reply → B 不回 5min 自动建 issue 升级
  • 多群并发 = 单 Agent maxConcurrent(默认 2),不是全局 · src/executor/worker.ts · 西花同时在 c211307e 放任务 + 2fec433e 排查,江德福两边接活互不堵
  • Agent 也会被实测打脸 + 用硬数据重做 · 2fec433e 群:瓦力 07:26 判"非对象 TypeError",西花 07:32/07:33 两次实测反驳,瓦力用 V8 实测纠错 + 三方证据关掉 R2 风险 · 单 Agent 给不出这种"被打脸后用硬数据重做"的回路
设计原则 4 · 持续学习 = 沉淀的资产在 rotom 里
  • memory 落 memory 表 · scope ∈ {group, global} · 2fec433e 群:2273d7fd(label/desc 根因 · 群级);c211307e 群:6c046004(Codex rollout 去重 · global)· /api/memory/global 可查

  • 人 review 后 promote · memory promote 走 /api/memory/:id/promote,status pending → approved · 人未 review 不进下次 task prompt 注入(避免错误记忆污染)
  • 沉淀物直接喂回 rotom 自身 · 4a9c1eaa / f080e51e / 54ab4777 三个 rotom 自迭代群:沉淀的 skill 回头用来优化 rotom 行为 · 用 rotom 迭代 rotom 自身
