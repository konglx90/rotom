外部集成 · 嵌入其他工作流 · 与 mesh Agent 沟通

上面讲的全是 rotom 内部协作 —— 还有一类场景:你的 codex / claude / 别的 agent 在外面干别的活,过程中需要把 mesh 里的 agent 拉进来问一句、协作一把。skill/rotom-bus-host 就是干这个的。

场景 · 谁在外面?怎么拉 mesh?

你的 codex / claude 已经在做别的活(写代码 / 写报告 / 排查问题),中间卡了:这块业务 mesh 里的江德福熟、这步架构 mesh 里的瓦力懂。

3 步搞定:

① 调 rotom group create 建群 · ② 调 rotom group send --need-reply 提问 · ③ 调 rotom group new-messages --since 等回复

身份来自 ~/.rotom/executor.config.json · --as=<你> 切 · 走 Bash 调 rotom CLI

两种群模式(按场景二选一)

普通群 · 多 Agent 协作 · @ 触发 worker 自动接话 · 谁都能冒泡

→ rotom group create "<需求>" --agents a,b,c --message "@全体 ..."

单播群 (--a2a-direct) · 点对点 · 消息只入库不广播 · worker 不自动醒 · 只在 --need-reply 显式点名时起一手回复(A 发 → B 回 → 停)

→ rotom group create "<你和 B 的小窗口>" --agents 你,B --a2a-direct

沟通循环 · 一问一答

提问:rotom group send <gid> <target> "<问题>" --need-reply

master 自动补 @target(确保对方 worker 起来)+ 登记 requestId → asker。对方回复时 master 硬剥掉 @<asker> 再入库 —— 你的 worker 不会被回复触发,不会来回聊

等回复:bash skill/rotom-bus-host/scripts/poll-replies.sh <gid> --as <你>

默认 10 轮 × 30s = 5min(覆盖 ask-bridge 超时窗口)· 找到退出 0 · 超时退出 2

关键约束(踩坑就完蛋)

❌ 轮询太快(每秒)→ 30s 间隔足够

❌ 不带 --need-reply 直接 send → 正文没 @target,对方 worker SKIP,消息白发

❌ 把明确任务塞群消息 → 用 issue create --run 派单

❌ 群消息 5+ 轮没收敛 → 升级为 task issue

❌ 编造对方回答(group send 异步)→ 等 new-messages 拉到再说

❌ 写盘没挂 issue → 先 rotom issue list <gid> --status in_progress,没有就先建
