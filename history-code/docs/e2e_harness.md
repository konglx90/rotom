# E2E Harness — 七维度框架

AI Agent 要可靠地完成端到端交付，需要一个 **harness（执行框架）** 来约束和支撑它的行为。一个完整的 harness 需要覆盖七个维度，缺一不可。

这七个维度不是功能列表，而是一个闭环：执行环境承载运行 → 工具协议定义交互 → 上下文状态传递知识 → 编排串联流程 → 可观测性暴露内部状态 → 评估反馈校准输出 → 安全策略划定边界。

---

## 1. 执行环境（Execution Environment）

Agent 运行在什么环境中，决定了它能影响多大的范围。

核心问题：
- Agent 的文件系统边界在哪？
- 有没有资源限制（CPU、内存、时间）？
- 环境不满足时能否提前阻断？

三个层次：

| 层次 | 含义 | 例子 |
|------|------|------|
| 预检 | 执行前检查环境是否就绪 | 工作目录存在、项目类型识别、依赖安装 |
| 隔离 | 运行时的作用域边界 | 工作目录限制、文件系统沙箱、网络限制 |
| 资源控制 | 防止资源耗尽 | 超时、最大 token 数、并发限制 |

预检是成本最低的保障。E2ED 在 `environment.ts` 中做项目类型检测和工作目录校验，但还没有隔离和资源控制。

关键原则：**环境问题应在执行前暴露，而非执行中崩溃。**

---

## 2. 工具协议（Tool Protocol）

Agent 通过工具与外部世界交互。工具协议定义了"能用什么工具"和"工具输入输出的格式"。

核心问题：
- Agent 能调用哪些工具？有没有白名单/黑名单？
- 工具的输入输出是否有结构化契约？
- 不同角色（生产者 vs 评审者）的工具权限是否不同？

两个约束：

| 约束 | 含义 | 例子 |
|------|------|------|
| 权限控制 | 谁能用什么工具 | 评审者只读不写，生产者可读写 |
| 格式契约 | 工具输入输出的结构 | 评审报告必须包含 score + verdict + findings |

格式契约比权限控制更容易被忽视，但同样关键。如果生产者的输出格式不固定（方案是自由文本而非结构化 JSON），评审者就无法可靠地解析和评估。

关键原则：**工具权限按角色最小化，产物格式用 schema 校验而非 prompt 软约束。**

---

## 3. 上下文状态（Context & State）

Agent 是无状态的——每次调用都是全新对话。Harness 必须在调用之间持久化和传递状态。

核心问题：
- 上一轮的关键决策如何传递给下一轮？
- 多轮交互后上下文如何避免无限膨胀？
- 中断恢复时能否从正确的状态继续？

三类状态：

| 类型 | 存储位置 | 例子 |
|------|----------|------|
| 进度状态 | 数据库 | 当前状态、版本号、activeTask |
| 产物 | 文件系统 | 方案文档、代码、评审报告 |
| 决策上下文 | 需要补充 | 前轮评审反馈、关键约束、被否决的方案 |

前两类 E2ED 已有（SQLite + 文件系统产物目录）。第三类"决策上下文"目前缺失——每轮 prompt 从头构建，前几轮的评审意见、决策理由、被否决的方案没有结构化传递，agent 可能重复犯错。

关键原则：**不只是持久化进度，还要持久化决策理由，避免跨轮知识衰减。**

---

## 4. 编排（Orchestration）

单次 Agent 调用只能做一件事。编排负责把多步任务串联成完整流程。

核心问题：
- 流程的步骤和顺序是什么？
- 某步失败后怎么处理？重试还是终止？
- 有没有条件分支（评审通过 vs 不通过走不同路径）？

三种编排模式：

| 模式 | 含义 | E2ED 应用 |
|------|------|-----------|
| 线性 | A → B → C 固定顺序 | plan → review → code → review |
| 条件 | 根据上一步结果分支 | 评审通过 → 下一步，不通过 → fix 循环 |
| 并行 | 多步同时执行 | 暂无 |

E2ED 目前是线性 + 简单条件分支（fix 循环）。缺失的：
- 重试策略：agent 执行失败（网络、超时）时没有自动重试
- 并行能力：需求评审和环境检查可以并行但没有

关键原则：**编排不是只定义 happy path，还要定义每一步失败后的恢复策略。**

---

## 5. 可观测性（Observability）

Agent 执行是黑盒。可观测性让你看到盒子里发生了什么。

核心问题：
- Agent 执行到哪一步了？
- 每一轮消耗了多少资源（token、时间、成本）？
- 产出了什么，改了哪些文件？

三个层级：

| 层级 | 含义 | E2ED 现状 |
|------|------|-----------|
| 进度 | 当前状态和版本 | status 命令 ✅ |
| 度量 | 耗时、轮次、通过率 | metrics 计算 ✅ |
| 深度 | token 消耗、文件变更、成本 | 缺失 ❌ |

E2ED 有进度和度量，但缺少"深度"层：每轮 deliver/review 消耗了多少 token、产出了哪些文件变更。这些是用户自查和成本控制的关键数据。

关键原则：**不只是记录"做了什么"，还要记录"花了多少"。**

---

## 6. 评估反馈（Evaluation & Feedback）

Agent 产出质量不可预测。评估体系负责量化质量，反馈机制负责将评估结果转化为改进。

核心问题：
- 评审标准是什么？是否可量化？
- 评审结果如何反馈给生产者？
- 多轮评审之间有没有趋势追踪？

评估体系三要素：

| 要素 | 含义 | E2ED 现状 |
|------|------|-----------|
| 评分标准 | 每个维度的检查项和权重 | 四维各 25% ✅ |
| 结构化输出 | 统一的评审报告格式 | VERDICT_JSON ✅ |
| 反馈闭环 | 评审 → 修复 → 再评审 | --fix 循环 ✅ |

可以增强的：
- **验收标准回验**：需求中的验收标准是具体的，代码评审应逐条检查是否满足，而非笼统评分
- **跨轮对比**：v1 评审指出的问题在 v2 是否修复了，目前靠人判断
- **评分校准**：不同评审维度之间的评分一致性

关键原则：**评估不只是打分，还要能追溯到具体缺陷，并验证缺陷是否被修复。**

---

## 7. 安全策略（Security Policy）

Agent 有能力修改文件系统、执行命令、访问网络。安全策略划定边界。

核心问题：
- Agent 能写哪些文件？能执行哪些命令？
- 生产者和评审者的权限是否隔离？
- 敏感操作有没有审计记录？

三道防线：

| 防线 | 含义 | E2ED 现状 |
|------|------|-----------|
| 审批控制 | 写操作需人工确认 | rw_allow / r_allow ✅ |
| 角色隔离 | 生产者不能自评，评审者不能改 | prompt 约束，非强制 ⚠️ |
| 审计追踪 | 所有写操作的记录 | 缺失 ❌ |

角色隔离目前靠 prompt 软约束（"你不能修改任何内容"），而非工具层面的硬限制。在内部场景下可接受，但如果开放给外部用户，需要工具级强制。

关键原则：**安全策略不应依赖 prompt 约束，应在工具层强制执行。**

---

## 七维度与 E2ED 的映射

| 维度 | 已覆盖 | 待补充 |
|------|--------|--------|
| 执行环境 | 环境预检 | 资源限制、超时控制 |
| 工具协议 | 审批策略 | 产物 schema 校验、角色工具白名单 |
| 上下文状态 | 进度 + 产物持久化 | 决策上下文传递、上下文衰减管理 |
| 编排 | 线性流程 + fix 循环 | 重试策略、并行能力 |
| 可观测性 | 耗时 + 轮次统计 | token/成本追踪、变更摘要 |
| 评估反馈 | 评分 + verdict + fix 循环 | 验收标准回验、跨轮对比 |
| 安全策略 | 审批控制 | 角色权限硬隔离、审计日志 |
